---

## What This Technique Is

An attacker with a list of common passwords systematically tries each one against a service until one works. No vulnerability is exploited — the attacker is betting on weak or reused credentials. SSH is a common target because it provides direct shell access if successful.

---

## Attack

### Step 1 — Reconnaissance

Confirm SSH is open on the target:

    sudo nmap -sV -p 22 192.168.178.25

Result: port 22 open, OpenSSH 9.6p1 Ubuntu.

### Step 2 — Brute Force

    hydra -l root -P /usr/share/wordlists/rockyou.txt ssh://192.168.178.25 -t 4 -V -f

| Flag | Meaning |
|---|---|
| `-l root` | Target username |
| `-P rockyou.txt` | Password wordlist (~14 million entries) |
| `-t 4` | 4 parallel connection threads |
| `-V` | Verbose — print each attempt |
| `-f` | Stop on first successful login |

Hydra opens SSH connections to node01 and attempts each password in sequence. Each failed attempt causes sshd to write a line to `/var/log/auth.log`.

---

## Detection

### Log Generated on node01

Every failed attempt produces a line like this in `/var/log/auth.log`:

    Jun 20 16:30:23 node01 sshd[1234]: Failed password for root from 192.168.178.29 port 54321 ssh2

The Wazuh agent on node01 continuously tails this file and forwards each line to the Wazuh manager on node03.

### Rules Triggered

| Rule ID | Level | Description | Trigger |
|---|---|---|---|
| 5760 | 5 | sshd: authentication failed | Every failed login |
| 5763 | 10 | sshd: brute force trying to get access to the system | 8+ failures from same IP within 60 seconds |

Rule 5763 is a frequency correlation rule. It watches for rule 5760 firing repeatedly from the same source IP and escalates automatically. This is the key alert.

### Alert Fields (Wazuh Dashboard)

| Field | Value |
|---|---|
| `rule.id` | 5763 |
| `rule.level` | 10 |
| `data.srcip` | 192.168.178.29 |
| `agent.name` | node01 |

The attacker IP was correctly identified in `data.srcip`.

---

## Why This Was Detected

sshd logs every authentication failure by default. Wazuh's agent collects these and the manager correlates repeated failures from the same source. Detection required no special configuration — it worked out of the box.

### What Was Not Detected

The preceding nmap port scan generated no alerts. Wazuh is log-based and cannot see network traffic directly. A network sensor (Suricata, Snort) would be needed to detect scanning.

---

## Defensive Controls

| Control | Notes |
|---|---|
| Disable password authentication | `PasswordAuthentication no` in sshd_config — already configured in this lab |
| Disable root SSH login | `PermitRootLogin no` in sshd_config |
| Wazuh active response | Auto-block attacker IP via iptables when rule 5763 fires |
| Fail2ban | Bans IPs after N failed attempts at the OS level |

Note: because password authentication is disabled on lab nodes, this attack would fail in practice. The simulation was run to validate the detection pipeline.

---

## Detection Tuning Notes

The default threshold for rule 5763 is 8 failures in 60 seconds. In environments with many users this may produce false positives from mistyped passwords. The threshold can be raised in a custom rule override, or exceptions can be added for known management IPs.

A slow brute force (one attempt per minute) would not trigger 5763. Detecting slow attacks requires a longer time window or integration with threat intelligence to flag known bad IPs on first attempt.
