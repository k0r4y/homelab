## What This Technique Is

After gaining initial access to a system with a low-privilege account, an attacker attempts to escalate to root using sudo. This simulation tests whether repeated sudo failures — which indicate an attacker probing privilege escalation paths — are detected by the SIEM.

---

## Attack

### Step 1 — Gain Initial Access

SSH into node01 as a standard user:

    ssh k0r4y@192.168.178.25

### Step 2 — Attempt Privilege Escalation

Deliberately enter incorrect passwords at the sudo prompt to trigger authentication failures:

    sudo su

Enter wrong passwords three times when prompted. sudo will output:

    [sudo] password for k0r4y:
    Sorry, try again.
    [sudo] password for k0r4y:
    Sorry, try again.
    [sudo] password for k0r4y:
    sudo: 3 incorrect password attempts

After three failures sudo exits and logs the event.

---

## Detection

### Log Generated on node01

sudo writes the following to the system journal (forwarded to `/var/log/auth.log`):

    Jun 20 15:34:00 node01 sudo[45223]: k0r4y : 3 incorrect password attempts ; TTY=pts/0 ; PWD=/home/k0r4y ; USER=root ; COMMAND=su

The Wazuh agent forwards this to the manager on node03.

### Rule Triggered

| Rule ID | Level | Description |
|---|---|---|
| 5404 | 10 | sudo: Three failed attempts to run sudo |

### Alert Fields (Wazuh Dashboard)

| Field | Value |
|---|---|
| `rule.id` | 5404 |
| `rule.level` | 10 |
| `data.srcuser` | k0r4y |
| `data.dstuser` | root |
| `data.tty` | pts/0 |
| `data.command` | su |
| `agent.name` | node01 |
| `manager.name` | node03 |

The full log line is preserved in `full_log`, giving analysts everything needed to understand what happened and from which terminal session.

---

## Why This Was Detected

sudo logs all authentication events including failures. The log entry contains the username, target user, working directory, TTY, and command attempted — enough context for an analyst to determine whether this was an accident or an attack.

Rule 5404 fires on a single event (three failures in one sudo invocation), unlike the SSH brute force rule which requires repeated attempts across multiple connections.

---

## Defensive Controls

| Control | Notes |
|---|---|
| Restrict sudo access | Limit sudoers to specific commands rather than ALL |
| Alert on any sudo failure | Rule 5404 level 10 is appropriate — consider paging on this in production |
| Audit sudo usage | Log all sudo invocations, not just failures, to establish baseline |

---

## Detection Tuning Notes

In a production environment with many developers using sudo regularly, failed sudo attempts from fat-fingering passwords are common and may generate noise. Consider:

- Suppressing rule 5404 for known admin users during business hours
- Escalating severity if the same user fails sudo more than once in a session
- Correlating with prior failed SSH logins from the same source to identify post-compromise escalation attempts
